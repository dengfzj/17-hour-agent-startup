# 2026-06-14 Update: API-Only Local Development and Bilingual UI

Language: [中文](./update-2026-06-14-zh.md) | **English**

Recorded on: 2026-06-14

## Summary

This update clarifies the local development split between the Vite frontend and the Express API, adds runtime Chinese/English switching in the application UI, and refreshes the public documentation so the startup instructions match the current scripts.

## What Changed

### 1. Local API mode is now explicit

`npm run api` now starts the backend in API-only mode:

```bash
npm run api
```

Default API address:

```text
http://127.0.0.1:8787
```

Health check:

```text
http://127.0.0.1:8787/api/health
```

In local API-only mode, opening `http://127.0.0.1:8787/` is expected to return an API 404 page. That is intentional. The frontend development entry remains Vite on port 5173.

### 2. Vite development frontend uses an API proxy

`npm run dev` starts the frontend at:

```text
http://127.0.0.1:5173
```

The Vite config proxies `/api/*` requests to the local Express API at `http://127.0.0.1:8787`, so frontend code can use same-origin `/api` paths during development.

### 3. Full local development command

Use one command for the normal local workflow:

```bash
npm run dev:full
```

This runs both:

- Vite frontend: `http://127.0.0.1:5173`
- Express API: `http://127.0.0.1:8787/api/health`

### 4. Production-style single-service mode remains supported

For a production-style local run:

```bash
npm run build
npm run start
```

In this mode the backend serves the built frontend from `dist/` and exposes the API from the same service. This behavior is separate from `npm run api`, which is intentionally API-only.

### 5. Runtime Chinese/English UI switching

The app now includes a global language toggle:

- main workspace pages show the toggle in the top bar;
- public checkout, legal, onboarding, and recovery pages show a fixed language toggle;
- language preference is saved in `localStorage`;
- `html lang` and the browser title update with the current language;
- user-entered business data, CSV content, emails, URLs, and numeric values are not rewritten by the localization layer.

The implementation keeps source files UTF-8 and relies on the existing UTF-8 declaration in `index.html` to avoid mojibake.
